
Brief explanation
GDB-UI is a modern, web-based interface for the GNU Debugger (GDB), designed to simplify the debugging process for developers working with C, and C++. It provides real-time interaction with GDB, enabling features like monitoring program execution, inspecting variables, setting breakpoints, and more, all through an intuitive web application.

GDB-UI enhances the debugging workflow by offering a sleek, user-friendly UI, replacing the traditional command-line experience with a visual and accessible alternative. It supports both Docker-based and manual setups, allowing seamless integration into various development environments.

Here’s a detailed technical report on how the pygdbmi package works internally and how it interfaces with GDB.
1. Overview
pygdbmi is a Python package that provides a programmatic interface to GNU Debugger (GDB) using its Machine Interface (MI) protocol rather than the traditional CLI interface.
Instead of parsing human-readable console output, pygdbmi communicates with GDB using structured MI commands and responses, making it suitable for:
Debugger frontends
IDE integrations
Automation tools
Test frameworks
Embedded debugging tools
It acts as a bidirectional bridge:
Python code  ⇄  pygdbmi  ⇄  GDB (MI mode)  ⇄  Debugged program

2. Core Architecture
pygdbmi is built around two primary components:
2.1 GDBController
This is the main class responsible for:
Launching GDB as a subprocess
Sending MI commands
Reading raw output
Parsing responses into structured Python dictionaries
It typically launches GDB like this:
gdb --interpreter=mi2

The --interpreter=mi2 flag enables Machine Interface version 2.
Internally, it uses Python’s:
subprocess.Popen
Pipes for stdin/stdout/stderr
Non-blocking reads (select or polling depending on platform)
2.2 MI Parser
The parser converts raw GDB MI output lines into structured Python objects.
MI output is structured but not JSON. It follows this grammar:
[token] "^" result-class ( "," result )*
[token] "*" async-class ( "," result )*
[token] "=" notify-class ( "," result )*
"~" console-output
"&" log-output

Example raw output:
^done,bkpt={number="1",type="breakpoint",disp="keep"}

pygdbmi parses this into:
{
  "type": "result",
  "message": "done",
  "payload": {
      "bkpt": {
          "number": "1",
          "type": "breakpoint",
          "disp": "keep"
      }
  },
  "token": None
}

3. GDB MI Protocol Explained
GDB Machine Interface has five main response types:
Prefix
Meaning
^
Result Record (synchronous command response)
*
Exec Async Record
=
Notify Async Record
~
Console Stream Output
&
Log Stream Output
pygdbmi detects these prefixes and categorizes them as:
"result"
"notify"
"console"
"log"
"target"
4. Internal Workflow
Step 1: Launching GDB
When GDBController() is instantiated:
It creates a subprocess running:
gdb --interpreter=mi2

It opens pipes for:
stdin (write commands)
stdout (read responses)
Step 2: Sending Commands
Commands are sent as MI commands:
controller.write('-break-insert main')

Under the hood:
The command is written to stdin
A newline is appended
Optionally a numeric token is prepended
Example actual transmitted command:
1-break-insert main

Token 1 helps match responses.
Step 3: Receiving Output
GDB outputs one or more lines:
Example:
^done,bkpt={number="1"}
(gdb)

pygdbmi:
Reads raw lines from stdout
Buffers partial lines
Splits at newline boundaries
Sends complete lines to the parser
Step 4: Parsing MI Output
The parser:
Identifies record type via first character
Extracts token (if present)
Parses:
key=value pairs
nested tuples
lists
strings with escaped characters
It builds structured Python dictionaries recursively.
5. Token System
Tokens are optional numeric prefixes attached to commands:
15-break-insert main

Response:
15^done

pygdbmi uses tokens to:
Match async responses to specific commands
Support concurrent command issuing
Avoid race conditions
If no token is supplied, None is used.
6. Data Structure of Responses
Each parsed response is returned as a dictionary:
{
    'type': 'result',
    'message': 'done',
    'payload': {...},
    'token': 1
}

Types:
result
console
log
notify
target
Multiple dictionaries may be returned per command.
7. Handling Asynchronous Events
GDB emits asynchronous events:
Example:
*stopped,reason="breakpoint-hit"

pygdbmi parses this as:
{
    "type": "notify",
    "message": "stopped",
    "payload": {
        "reason": "breakpoint-hit"
    }
}

These are critical for:
Breakpoints
Program stop events
Thread creation
Signals
The library does not block waiting for only one response; it collects all output until timeout.
8. Timeout and Blocking Behavior
write() supports:
timeout_sec
Non-blocking reads
Mechanism:
After writing a command, it polls stdout
Collects output until:
Timeout reached
Or no more output available
This makes it suitable for event-driven systems.
9. Example End-to-End Flow
Python Code
from pygdbmi.gdbcontroller import GDBController

gdbmi = GDBController()

gdbmi.write('-file-exec-and-symbols a.out')
gdbmi.write('-break-insert main')
gdbmi.write('-exec-run')

Under the Hood
Launch GDB MI
Send commands via stdin
Receive MI formatted output
Parse output to dictionaries
Return structured results
10. Error Handling
If GDB reports an error:
Raw:
^error,msg="No symbol table loaded."

Parsed:
{
  "type": "result",
  "message": "error",
  "payload": {
      "msg": "No symbol table loaded."
  }
}

User code must inspect "message" field.
11. Internal Parser Complexity
The parser must handle:
Nested dictionaries
Lists inside dictionaries
Escaped strings
Quoted values
Tuple syntax
Example MI snippet:
frame={level="0",func="main",args=[{name="argc",value="1"}]}

Parser builds nested Python structures recursively.
It does NOT use JSON parsing — it implements a custom recursive descent parser tailored to MI grammar.
12. Platform Considerations
Works on Linux, macOS, Windows
Uses cross-platform subprocess handling
Some edge cases exist for Windows newline handling
13. Limitations
It does not implement full GDB state modeling
It is a thin wrapper — no high-level debugging abstractions
Requires external GDB installation
No built-in async event loop integration (user must manage)
14. Comparison to Alternatives
pygdbmi differs from:
CLI wrappers (string parsing)
Direct GDB Python API (embedded inside GDB)
Full debugger frameworks
It focuses specifically on MI-level automation.

1. Objective
The goal is to transform the current experimental Docker-based GDB sandbox into a fully persistent, state-driven remote debugger platform.
Each user session will:
Run inside an isolated Docker container

Maintain one persistent pygdbmi.GdbController

Expose structured debugger state (threads, stack, locals, registers, memory, breakpoints, functions)

Automatically synchronize UI state after every debugger event

Remain reconnectable without losing debugger context

This design ensures IDE-grade debugging behavior while maintaining strict sandbox isolation.

3. Core Architectural Components

3.2 Persistent GdbController per Container
For each session:
A Docker container is created with resource and security limits

GDB is launched via:

docker exec -i <container> gdb --interpreter=mi3 ./binary

The pygdbmi.GdbController is instantiated on the host with:
gdb_path = "docker"

This ensures:
A single persistent GDB instance per container

Proper MI protocol handling

No raw terminal parsing

Structured response parsing via pygdbmi

The controller lives for the entire lifetime of the debugging session.
4. Canonical Debugger State Model
Instead of streaming raw MI output to the frontend, the backend maintains an authoritative state model.
The DebugState object includes:

Stack frames

Local variables

Global variables

Registers

Breakpoints

This state model:
Enables reconnection without losing debugger context

5. MI Event Processing Model
A dedicated background thread continuously polls:
controller.get_gdb_response()

All MI responses are classified into:
result

exec

notify

console

log

target

Each response type updates the internal state model.
For example:
exec → stopped triggers a full state synchronization

notify → thread-created updates thread list

result → stack updates stack frames

result → locals updates local variables

This ensures accurate synchronization with the debugger’s real state.
6. Automatic State Synchronization
To emulate an IDE-grade debugger, the backend automatically synchronizes all relevant state after:

-thread-info

-stack-list-frames

-stack-list-locals 1

-data-list-register-values x

-break-list

-info-functions

-interpreter-exec console "info proc mappings"

This guarantees that:

Local variables are updated

Registers are refreshed

The UI updates automatically after receiving the new state snapshot.

Session Management for Multiuser Support( Consider S3 for persistent storage)
Objective
The system must support multiple concurrent users, each with an isolated, persistent debugging environment.
The goals are:
Each user session must be completely isolated from others.

Debugger state must persist for a defined duration.

No user must be able to access another user's files, processes, or memory.

Session lifecycle must be managed automatically.

Persistence duration must vary based on authentication level.

To achieve this, the platform uses Docker-based sandboxing combined with a structured backend session manager.
1. Per-User Debugging Sessions
1.1 Unique Session Identification
Each debugging session is assigned a unique session_id (UUID).
This ID is used to:
Create a dedicated workspace directory

Spawn a dedicated Docker container

Associate a persistent GdbController

Map WebSocket connections to the correct backend session

Internally:
session_id → DebugSession

This ensures that all debugger operations are strictly scoped to the correct container.
2. Docker-Based Isolation Model
The foundation of multiuser isolation is one Docker container per debugging session.
For each user:
A new container is created.

A workspace directory is mounted exclusively for that session.

A persistent GDB instance runs only inside that container.

No container shares filesystem, network, or process space with another.

2.1 Filesystem Isolation
Each session gets:
/tmp/sandboxes/<session_id>/

This directory is:
Mounted into the container as /workspace

Not shared across sessions

Not accessible outside its container

The container root filesystem is:
Read-only (except /workspace)

Minimal and hardened

Does not expose host directories

This guarantees:
No cross-user file access

No ability to inspect other users' source code

No modification of host filesystem

2.2 Process Isolation
Each container:
Has its own PID namespace

Cannot see processes from other containers

Cannot access host processes

Runs under a non-root user

Even if malicious code executes inside one container, it cannot:
Attach to another session’s process

Access another container’s memory

Escalate privileges outside its namespace

2.3 Network Isolation
Containers are launched with:
network_mode = none

This ensures:
No internet access

No ability to scan local network

No communication between containers

This prevents:
Malware attempts

Data exfiltration

Cross-session attacks

2.4 Resource Isolation
Each container enforces:
Memory limits

CPU limits

Process limits

File descriptor caps

This ensures that one user cannot exhaust system resources and degrade service for others.
3. Persistent Debugger State per User
Each container has exactly one persistent GdbController.
This ensures:
Breakpoints remain intact

Stack context remains intact

Thread state persists

Memory and register values are preserved

Reconnection restores full debugging context

If a WebSocket disconnects temporarily:
The container remains alive

The controller remains active

The user can reconnect without losing state

4. Session Persistence Strategy
Persistence duration depends on authentication level.
4.1 Non-Authenticated Users (Temporary Persistence)
For anonymous users:
A session is created immediately.

Workspace is mounted at /tmp/sandboxes/<session_id>.

Session remains active while WebSocket is connected.

After disconnection:

Session persists for a short grace period (e.g., 10–30 minutes).

If not reconnected → container is stopped.

Workspace directory is deleted.

This provides:
Temporary persistence

Protection from resource leakage

Automatic cleanup

Minimal disk consumption

4.1.1 Expiration Mechanism
A background cleanup task runs periodically:
Checks last_activity_timestamp

If inactivity exceeds threshold:

Stop container

Destroy GdbController

Delete workspace directory

This prevents accumulation of unused sessions.
4.2 Authenticated Users (Extended Persistence)
For authenticated users:
Sessions are tied to a user account.

Workspace is stored in a longer-term mounted directory, e.g.:

/data/users/<user_id>/<project_id>/

Instead of /tmp.
This enables:
Long-term project storage

Debugging history preservation

Persistent breakpoints across sessions

Project-based workflows

Session persistence rules:
Containers shut down after inactivity.

Workspace remains intact.

When user returns:

Container is recreated

GdbController reinitialized

Breakpoints and project files restored

This model allows:
Stateless backend processes

Stateful project storage

Controlled resource usage

5. Saving Debugging Progress
Progress includes:
Source files

Compiled binaries

Breakpoints

Optional session metadata (last file, cursor position, etc.)

5.1 File Saving
The editor continuously syncs code changes to the mounted workspace.
Since the workspace is mounted:
File updates are immediately persisted on host disk.

No additional save step is required.

5.2 Breakpoint Persistence
Before session shutdown:
Breakpoints can be exported from GdbController.

Stored in a metadata JSON file in workspace.

Example:
{
 "breakpoints": [
   {"location": "main"},
   {"location": "solve"}
 ]
}

On session recreation:
Breakpoints are reinserted automatically.

5.3 Optional Snapshot Feature (Future Enhancement)
For authenticated users:
Allow explicit “Save Snapshot”

Store:

Source files

Breakpoints

Execution state (frame info)

Enable project restoration later

6. Reconnection Mechanism
When a user reconnects:
Frontend sends session ID.

Backend checks:

If session active → reattach to container and controller.

If container stopped but workspace exists → recreate container.

If expired → start new session.

For authenticated users:
Session mapping stored in database.

Multiple projects per user supported.

7. Preventing Cross-User Interference
The following guarantees isolation:
Layer
Isolation Guarantee
Docker container
Separate namespaces
Filesystem mount
Unique per session
Network
Disabled
Resource limits
Enforced per container
GdbController
One per session
Session registry
No shared state
There is no shared:
Process

File

Memory

Debugger instance

This eliminates interference entirely.
8. Scalability Considerations
Since each user consumes:
One container

One GdbController

Some file descriptors

Some memory

The host must be configured with:
Raised file descriptor limits

Sufficient memory

Monitoring for concurrent sessions

The architecture scales linearly with resources.
9. Summary
This session management architecture ensures:
True multiuser support

Complete isolation via Docker

Persistent debugging state

Temporary storage for anonymous users

Long-term project persistence for authenticated users

Automatic cleanup to prevent resource exhaustion

No cross-user interference

The system combines:
OS-level isolation (Docker namespaces)

Structured backend session management

Controlled persistence policies

Reactive debugger architecture

This creates a secure, scalable, multiuser debugging platform suitable for educational and production environments.



Real-Time Debugging Results
Objective
The system is designed to provide real-time debugger updates without requiring page refreshes.
This is achieved using a persistent WebSocket connection between the frontend and backend, combined with an event-driven GDB/MI processing layer.
The architecture guarantees:
Immediate UI updates on debugger state changes

Low-latency interaction

No polling overhead

Efficient bidirectional communication

Scalable multiuser support

1. High-Level Architecture
Each active debugging session consists of:
Frontend (Web UI)
   ⇅ WebSocket
FastAPI Backend (Session Manager)
   ⇅
Persistent GdbController
   ⇅
Docker Sandbox Container (GDB running inside)

The WebSocket remains open for the entire session lifetime.
2. WebSocket Lifecycle
2.1 Connection Establishment
Frontend connects to:
wss://server/ws/debug/{session_id}

Handshake steps:
Client sends WebSocket upgrade request.

Backend validates:

Session ID

Authentication (if required)

Backend associates WebSocket with:

Existing DebugSession

Existing GdbController

Backend starts event streaming loop.

At this point, the session becomes fully interactive.
2.2 Connection Binding
Internally:
session_id → DebugSession
DebugSession:
   - container_id
   - gdb_controller
   - websocket
   - state_snapshot
   - event_queue

Each WebSocket maps strictly to one session.
No shared connections.
3. WebSocket Message Protocol
The protocol is structured, JSON-based, and event-driven.
There are two categories of messages:
Client → Server (Commands)

Server → Client (Events)

4. Client → Server Protocol
Each client message follows this structure:
{
 "type": "command",
 "command": "step_over",
 "payload": {}
}

General schema:
{
 "type": "<category>",
 "command": "<action>",
 "payload": { ... },
 "request_id": "<optional-correlation-id>"
}
4.1 Supported Command Types
Execution Commands
run

continue

step_into

step_over

step_out

interrupt

Breakpoint Commands
{
 "type": "command",
 "command": "add_breakpoint",
 "payload": {
   "file": "main.cpp",
   "line": 42
 }
}
Inspection Commands
get_stack

get_locals

get_threads

get_registers

get_memory

5. Backend Processing Flow
When a command is received:
WebSocket handler parses message.

Retrieves DebugSession via session_id.

Translates command into GDB/MI instruction.

Sends MI command to persistent GdbController.

Waits for MI response.

Updates internal DebugState.

Emits structured event to client.

6. GDB/MI Event Listener Loop
The backend runs a dedicated background task per session:
async def gdb_event_listener():
   while session.active:
       responses = gdb_controller.get_gdb_response()
       parse_responses(responses)
       update_debug_state()
       push_websocket_updates()

This loop captures:
Breakpoint hits

Program stops

Thread creation

Signal events

Execution completion

This enables reactive UI updates.
7. Server → Client Event Protocol
All backend updates are pushed automatically.
Message structure:
{
 "type": "event",
 "event": "state_update",
 "data": {
   "stack": [...],
   "locals": {...},
   "threads": [...],
   "breakpoints": [...]
 }
}

1️⃣ Execution State Updates

Program stops

Breakpoint hit

Step completes

Signal received

 "type": "event",

 "data": {
   "reason": "breakpoint-hit",
   "location": {
     "file": "main.cpp",
     "line": 42
   }
 }
}
2️⃣ Stack Updates
{
 "type": "event",
 "event": "stack_updated",
 "data": {
   "frames": [...]
 }

3️⃣ Locals Update

 "type": "event",

4️⃣ Thread Updates
Triggered on:

Thread switch

GDB console output:

8. State Synchronization Model
Instead of refreshing the page:
The backend maintains a canonical DebugState.

After any MI event:

State is updated.

Relevant deltas are pushed.

Frontend updates only affected UI components.

This avoids full UI re-render.
9. Avoiding Polling
Traditional systems use polling:
GET /stack
GET /locals
GET /threads

This is inefficient.
In this architecture:
All updates are push-based.

No repeated HTTP calls.

No redundant queries.

WebSocket ensures:
Persistent connection

Near-zero latency updates

Reduced server load

10. Concurrency and Isolation
Each session has:
Dedicated WebSocket

Dedicated GdbController

Dedicated event loop

There is no shared event bus across users.
Thus:
One user stepping does not trigger events for others.

No race conditions across sessions.

If an error occurs:

 "type": "error",

Frontend can react appropriately.
12. Heartbeat Mechanism
To detect stale connections:
Client sends ping every N seconds.

Server responds with pong.

If no heartbeat:

Session marked inactive.

Grace timeout begins.

This prevents zombie WebSockets.

If WebSocket disconnects:
Session remains alive.

Server resends full state snapshot.

UI reconstructs entire debugging state.
